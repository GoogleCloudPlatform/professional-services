# Performance Tuning & Latency Optimization

This document details the features and strategies used by the Eval Pipeline to achieve high performance and low latency, matching or exceeding ad-hoc automation scripts.

## 1. Context Management (Screenshot Scrubbing)

The primary cause of latency in long-running browser tasks is **Context Bloat**. As the conversation history grows, the number of tokens sent to the model increases, leading to slower prediction times (spiking from 5s to 30s+).

### Screenshot Scrubbing
To mitigate this, the pipeline implements **Screenshot Scrubbing**.
*   **What it does:** Automatically removes image data from older turns in the history.
*   **What stays:** The screenshot from the **first turn** (initial state) and the **latest turns** are preserved.
*   **Configuration:**
    Use the `context` presets in your benchmark YAML.
    ```yaml
    agent:
      context:
        preset: "BALANCED" # Keep last 5 images (Default)
        preset: "EFFICIENT" # Keep last 3 images
    ```

### History Summarization
For extremely long conversations, the pipeline condenses the middle history.
*   **What it does:** Replaces the middle turns of the history with a semantic text summary generated by a fast model.
*   **Safety:** This **never** touches the **Goal** or the **Instruction Plan** (first 3 turns).
*   **Benefit:** Keeps the "turn count" low for 100+ step tasks.
*   **Configuration:** Enable via `preset: "EFFICIENT"`.

## 2. Action Efficiency (Smart Batching)

Network round-trips are expensive. The pipeline uses **Smart Batching** to execute multiple actions in a single turn without sacrificing grounding or reliability.

### Terminal Slicing
To ensure the agent doesn't act on stale information, the `ToolExecutor` automatically truncates batches if a **Terminal Action** occurs. 
*   **Terminal Actions:** `navigate`, `go_back`, `KeyCombination(Enter)`, and `type_text_at(press_enter=True)`.
*   **Benefit:** The agent is forced to "re-observe" the page immediately after a major state change (like a page load), preventing blind clicks on the wrong page.

### Fast Path Bundling (Input Speedup)
Consecutive text-entry actions (`type_text_at`) are automatically bundled into a single high-speed execution.
*   **How it works:** Instead of simulating individual keypresses for each field, the system injects the text into the DOM directly using `page.evaluate`.
*   **Impact:** Reduces input latency by **~90%** for form-filling tasks.

### Perception Guards
Between every action in a batch, the system checks for **Significant UI Shifts**.
*   **Check:** If the URL changes OR the number of interactive elements in the ARIA tree changes by >20%, the remaining actions in the batch are cancelled.
*   **Result:** The agent receives a `CANCELLED_BY_PERCEPTION_GUARD` result for the skipped actions, prompting it to re-evaluate the new state.

## 3. Viewport Parity & Coordinate Integrity

A mismatch between the agent's viewport and the target application's design can cause "Navigation Blindness."

*   **CoordinateScaler:** The pipeline uses a centralized `CoordinateScaler` (0-1000 scale) to ensure that model-generated coordinates are perfectly mapped to the browser's pixel space, regardless of the resolution or device pixel ratio.
*   **Clamping:** Gemini may occasionally generate coordinates slightly outside the `[0, 1000]` range. The `CoordinateScaler` automatically **clamps** these values to the valid range to prevent `400 INVALID_ARGUMENT` crashes.
*   **Parity Resolution:** Always run benchmarks at the resolution the application was designed for (e.g., `1440x900` for many ERP tools).

## 4. Wait Discipline (Latency vs. Reliability)

Excessive waiting is the enemy of speed. If an agent waits 5 seconds after every click "just to be safe", a 50-step task gains **4 minutes** of idle time.

*   **System Micro-Waits:** The underlying Playwright engine automatically waits for elements to be stable and actionable before clicking/typing. You generally do **not** need explicit waits for standard UI interactions.
*   **When to avoid explicit waits:**
    *   **Long Processes:** The framework detects network idle automatically.
    *   **Visual Confirmation:** The agent will naturally observe the next frame.
*   **Prompting Strategy:** Instruct the agent to trust the system's micro-waits for simple navigation (tabs, dropdowns, checkboxes).

## 5. Understanding Telemetry

The pipeline logs granular timing and token usage for every step. Look for these lines in your logs:

```text
⏱️  [TELEMETRY] Step 34 Duration: 11.56s (Total: 321.00s)
   Context Prep: 0.01s
   Middleware:   0.04s
   Pre-Observe:  0.00s
   Prediction:   5.89s
   Execution:    5.55s
   Post-Observe: 0.08s
📊 [TOKENS] Total Input: 237516 | Total Cached: 78269 | Total Output: 3381
```

### Latency Breakdown
*   **Context Prep:** Time spent running the Context Pipeline (Compaction, Summarization, Trimming) to prepare the history for the model.
*   **Middleware:** Time spent running safety checks and reflection/stalemate detection.
*   **Pre-Observe:** Initial state capture before the model acts.
*   **Prediction:** Time spent waiting for the Gemini model to return actions (the most expensive phase).
*   **Execution:** Time spent running the Playwright actions (typing, clicking, scrolling).
*   **Post-Observe:** Time spent capturing the new screenshot and DOM state after execution.

### Token Tracking
*   **Total Input:** The cumulative sum of all tokens sent to the model across all steps.
*   **Total Cached:** The cumulative sum of tokens that were successfully served from Vertex AI Context Caching. A high ratio of Cached to Input tokens means your context pipeline is well-optimized.
*   **Total Output:** The cumulative sum of tokens generated by the model (action payloads and reasoning).

### Latency Targets
*   **Optimal:** Prediction < 8s per step.
*   **Degraded:** Prediction > 20s per step (indicates context bloat, broken caching, or network issues).
