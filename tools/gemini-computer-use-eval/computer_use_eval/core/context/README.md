# Context Pipeline: Preventing Context Rot

Multimodal agents (like those using Computer Use) generate massive amounts of context per turn due to screenshots and DOM state. Without active management, these agents quickly reach token limits or suffer from "Context Rot"—where the model loses track of the primary goal due to the sheer volume of intermediate data.

This directory contains a pluggable **Context Pipeline** that optimizes conversation history before it is sent to the LLM, ensuring high performance over long horizons.

## Core Concepts

The pipeline uses three distinct strategies to reduce context while preserving execution integrity. We prioritize **Reversibility**, **Momentum**, and **API Validity**.

### 1. Compaction (Reversible/Lossless-ish)
Compaction strips out redundant information that exists in the environment or can be mathematically condensed.
- **Loop Compression:** Detects repetitive actions (e.g., a sequence of 10 `scroll` or `wait` calls) and replaces them with a single text summary like `[Summarized: Agent executed 'wait' 10 times to stabilize UI/Navigation.]`.
- **Screenshot Scrubbing:** Removes heavy image data from intermediate turns based on fidelity configurations (e.g., keeping only the Turn 0 Goal and the most recent 3 turns).

### 2. Summarization (Lossy, Token-Triggered)
When the history becomes too heavy, the pipeline uses an LLM to condense the "middle" of the conversation.
- **The Token Trigger:** Summarization is triggered if the token count exceeds the `summarization_token_threshold` (e.g., 80k tokens). This ensures the history is gracefully compressed *before* it gets too large for the context window.- **Head & Tail Preservation:** The pipeline always preserves the **Head** (Turn 0: The Goal) and the **Tail** (Last N turns: Momentum) in their raw, full-detail format. 
- **Middle Narrative:** The intermediate steps are replaced with a concise narrative summary generated by the LLM.

### 3. Smart Trimming (Safety Fallback)
As a final layer of protection, the pipeline implements a "Head-Tail" trim that removes the oldest intermediate turns if they still exceed the token limit after compaction and summarization.

## Architectural Highlights

The Context Pipeline handles several complex edge cases specific to the Gemini API's strict validation rules.

### Orphaned Tool Responses (ADK Inspired)
When slicing history into Head, Middle, and Tail, it is critical not to split a `function_call` from its `function_response`. Our pipeline uses a backward-iterating set-tracking algorithm (inspired by Google ADK's `_safe_token_compaction_split_index`) to ensure no unmatched tool responses are left hanging in the Tail if their originating calls get pushed into the compacted Middle.

### Robust Role Bridging
The Gemini API enforces a strict alternating role policy (`User -> Model -> User -> Model`). When the pipeline cuts out the middle of the history, the end of the Head and the start of the Tail might clash (e.g., `User` followed by `User`).
Instead of silently deleting turns to fix the clash (which destroys data), the pipeline dynamically injects **Bridge Turns**:
- In **Summarization**, it injects a perfectly balanced sequence (e.g., `[Model(Summary), User("Acknowledged.")]`) to stitch the Head and Tail together flawlessly.
- In **Smart Trim**, it injects a dummy `[History Trimmed]` turn to safely bridge the gap without dropping the agent's momentum.
- If an LLM summarization API call fails, the pipeline safely aborts and returns the *unmodified* history to prevent data loss.

## Configuration

You can configure the pipeline directly in your Benchmark YAML:

```yaml
agent:
  model: "gemini-3.0-flash-preview"
  
  context:
    preset: "EFFICIENT"                 # ACCURATE, BALANCED, or EFFICIENT
    enable_compaction: true             # Enable lossless loop folding
    enable_context_caching: true        # Enable Vertex AI context caching
    enable_summarization: true          # Enable LLM-based summarization
    summarization_token_threshold: 80000 
    max_images_in_history: 3            # Rolling window for screenshots
```

## Developer Guide: Adding a Strategy

The pipeline is designed to be extensible. To add a new strategy (e.g., PII scrubbing, HTML minification), implement the `ContextStrategy` interface:

```python
from computer_use_eval.core.context.base import ContextStrategy

class MyCustomStrategy(ContextStrategy):
    async def apply(self, history: List[types.Content]) -> List[types.Content]:
        # Implement your logic here
        # Return the modified history
        return history
```

## Why this is Non-Destructive
The framework maintains two versions of history:
1. **The Trace (`history.json`):** A full, immutable record of every raw turn, screenshot, and tool output. This is used for generating the final `result.json` and for evaluation debugging.
2. **The Context (The Projection):** A transient, optimized version created by the pipeline specifically for the next API call.

This ensures that while the model stays within token limits and API constraints, the **human evaluator** always has the full story.
